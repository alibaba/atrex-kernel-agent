# FlashInfer Integration of TensorRT-LLM Cubin Kernels

Technical analysis of how FlashInfer integrates TensorRT-LLM's optimized kernels via cubin dynamic loading, without importing the entire TRT-LLM source tree. Covers the MoE module as a case study.

---

## 1. Introduction

FlashInfer elegantly integrates TensorRT-LLM's optimized kernels into its own ecosystem without bundling the entire TRT-LLM source. This analysis uses the MoE module as an example. While TRT-LLM has many open-source standalone kernels (DeepSeekV3 low-latency GEMM, communication kernels, etc.) that are straightforward to integrate, this document focuses specifically on **closed-source kernels distributed only as cubin files**.

**Core problems solved:**

- How to use TRT-LLM's optimized kernels without directly depending on TRT-LLM source?
- How to dynamically load and manage large numbers of cubin files generated by the TensorRT-LLM Gen system?
- How to keep the source tree from becoming bloated?

## 2. TensorRT-LLM Gen System

TRT-LLM Gen is NVIDIA's code generation system that automatically produces highly optimized CUDA kernels. Key template fields:

- `TLLM_GEN_EXPORT_VERSION "6.0.3.0.2.1"` — version info
- `TLLM_GEN_COMMIT "c603ed2"` — commit hash
- `TLLM_GEN_BATCHED_GEMM_CONFIG_HASH "65deb07"` — config hash

Modules using the Gen system include GEMM / BatchGEMM / MoE GemmGatedAct, and MHA / MLA. These represent TRT-LLM's competitive moat — highly optimized and distributed only as cubin.

## 3. Integration Architecture

```
FlashInfer Python API
  ↓
JIT compilation layer (fused_moe.py)
  ↓
CUDA kernel launcher (trtllm_fused_moe_kernel_launcher.cu)
  ↓
TRT-LLM kernel runner (trtllm_fused_moe_runner.cu)
  ↓
TRT-LLM kernel implementation (DevKernel.h, RoutingKernel.h, etc.)
  ↓
CUBIN dynamic loading and execution
```

**Layered responsibilities:** Interface layer (PyTorch-compatible API) → Dispatch layer (kernel selection and parameter validation) → Execution layer (coordinating routing + MoE kernels) → Loading layer (dynamic cubin management).

## 4. Source Integration Techniques

### 4.1 Header File Integration

FlashInfer copies TRT-LLM's key header files directly into its own `include` directory:

```cpp
// trtllm_fused_moe_kernel_launcher.cu
#include "flashinfer/trtllm/batched_gemm/trtllmGen_bmm_export/trtllm/gen/DtypeDecl.h"
#include "flashinfer/trtllm/fused_moe/DevKernel.h"
#include "flashinfer/trtllm/fused_moe/RoutingKernel.h"
#include "flashinfer/trtllm/batched_gemm/trtllmGen_bmm_export/BatchedGemmInterface.h"
```

### 4.2 Namespace Isolation

Separate namespaces (`tensorrt_llm::kernels::trtllmGenFp8BlockScaleMoe`, `trtllm_cubin_loader`, etc.) prevent conflicts with FlashInfer's own kernels.

### 4.3 Core Source Files

| File | Function | Role |
|------|----------|------|
| `trtllm_fused_moe_kernel_launcher.cu` | PyTorch interface layer | Parameter validation, memory management, kernel dispatch |
| `trtllm_fused_moe_runner.cu` | Fused MoE runner | Coordinates routing and MoE execution |
| `trtllm_fused_moe_routing_kernel.cu` | Routing kernel | Expert selection and load balancing |
| `trtllm_fused_moe_dev_kernel.cu` | Device kernel | Low-level CUDA implementation |
| `trtllm_batched_gemm_runner.cu` | Batched GEMM runner | Matrix multiplication optimization |

## 5. CUBIN Dynamic Loading System

### 5.1 Callback Mechanism

```cpp
void (*callbackGetCubin)(const char* path, const char* sha256) = nullptr;

extern "C" void FlashInferSetCubinCallback(void (*callback)(const char* path, const char* sha256));

std::string getCubin(const std::string& name, const std::string& sha256);
```

### 5.2 CUBIN Path Construction

```cpp
const std::string pipeline_hash = "39b7e49bfedde88ea29bfdc2547cbba659f2b236";
const std::string cubin_path = pipeline_hash + "/" +
    std::string("batched_gemm-") + TLLM_GEN_COMMIT + "-" +
    TLLM_GEN_BATCHED_GEMM_CONFIG_HASH + "/";
```

Path format: `{pipeline_hash}/batched_gemm-{TLLM_GEN_COMMIT}-{TLLM_GEN_BATCHED_GEMM_CONFIG_HASH}/{kernel_name}`. Example:

```
39b7e49bfedde88ea29bfdc2547cbba659f2b236/batched_gemm-c603ed2-65deb07/Bmm_Bfloat16_E2m1E2m1_Fp32_t128x16x256_et128x16_m128x16x64_cga1x1x1_16dp256b_s6_TN_transOut_schedP_bN_dynBatch_sm100a.cubin
```

Loading:

```cpp
auto fiModuleLoadData = [&](CUmodule* module) {
    const std::string sha256 = config.mHash ? config.mHash : "";
    // ... construct fname_cubin
    std::string cubin = flashinfer::trtllm_cubin_loader::getCubin(fname_cubin, sha256);
    cuModuleLoadData(&cuModule, cubin.c_str());
};
```

### 5.3 Remote Repository

```python
FLASHINFER_CUBINS_REPOSITORY = os.environ.get(
```

### 5.4 Caching Mechanism

```python
def get_cubin(name, sha256, file_extension=".cubin"):
    cubin_fname = name + file_extension
    cubin_path = FLASHINFER_CACHE_DIR / "cubins" / cubin_fname
    # 1. Try local cache
    cubin = load_cubin(cubin_path, sha256)
    if cubin:
        return cubin
    # 2. Download from remote repository
    uri = FLASHINFER_CUBINS_REPOSITORY + "/" + cubin_fname
    download_file(uri, cubin_path)
    return load_cubin(cubin_path, sha256)
```

## 6. JIT and Kernel Configuration

`setup_cubin_loader` registers the Python-side `get_cubin` as a C++ callback, passing cubin byte array pointers back to C++ via `FlashInferSetCurrentCubin`:

```python
def setup_cubin_loader(dll_path: str):
    _LIB = ctypes.CDLL(dll_path)
    def get_cubin_callback(name, sha256):
        cubin = get_cubin(name.decode("utf-8"), sha256.decode("utf-8"))
        _LIB.FlashInferSetCurrentCubin(
            convert_to_ctypes_char_p(cubin),
            ctypes.c_int(len(cubin))
        )
    cb = CALLBACK_TYPE(get_cubin_callback)
    _LIB.FlashInferSetCubinCallback(cb)
```

**Batched GEMM naming convention:** `Bmm_{InputType}_{WeightType}_{OutputType}_t{TileSize}_et{EpilogueTile}_m{MmaTile}_cga{ClusterGridArray}_{Stages}_{Transpose}_{Schedule}_{BatchMode}_{Architecture}`.

Example: `Bmm_Bfloat16_E2m1E2m1_Fp32` = input BF16 / weight E2m1 / output FP32; `t128x16x256` = tile size; `schedP` = parallel schedule; `bN` = batch N dimension; `sm100a` = SM100 architecture.

## 7. Summary

Through header file copying + namespace isolation + remote cubin repository + JIT callbacks, FlashInfer uses TRT-LLM Gen system's highly optimized kernels without importing TRT-LLM source. The source tree remains compact before JIT compilation.

Notable considerations:
- The TRT-LLM Gen kernel system has no official NVIDIA documentation analyzing its internals (closed-source)
- The cubin distribution approach is best suited for semiconductor vendors; for open-source teams, integrating the full solution adds significant weight
