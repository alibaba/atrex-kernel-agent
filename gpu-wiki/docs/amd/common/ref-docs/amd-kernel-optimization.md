## 1. AMD GPU Kernel Disassembly Analysis Methods

Understanding ISA through disassembly is a key approachhed to AMD GPU optimization. Unlike NVIDIA PTX, which describes the behavior of a single work-item, **AMDGPU assembly describes the behavior of an entire wavefront**.

### 1.1 Disassembly Workflow

Use ROCm's clang++ compiler to generate device assembly:

```bash
clang++ -x hip --offload-arch=gfx1103 --offload-device-only -O2 -S kernel.cpp
```

Reference documents:
- **LLVM AMDGPUUsage Documentation**: Assembly syntax, calling conventions, IR attributes
- **RDNA3/CDNA3 ISA Documentation**: Instruction set, hardware architecture overview

### 1.2 Key ISA Concepts

| Concept | Description |
|------|------|
| **SGPR vs VGPR** | SGPR is a scalar register shared by the wavefront; VGPR is a vector register unique to each work-item. The LLVM IR generated by the Clang frontend is at the work-item level; the AMDGPU back-end is responsible for assigning work-item-independent variables to SGPRs |
| **EXEC Mask** | `v_cmpx_gt` modifies the EXEC register to control which lanes execute. Branching is implemented via `s_cbranch_execz`—a jump occurs only when all work-items in the wavefront do not satisfy the condition |
| **s_waitcnt** | Waits for specific types of instructions to complete, appearing between loads and stores. Non-memory data dependencies are automatically tracked by hardware |
| **s_delay_alu** | A delay hint instruction introduced with RDNA3 to help the scheduler optimize instruction issue |

### 1.3 Special Instructions Requiring Manual Implementation

The following instructions cannot be directly expressed in C++ and require intrinsic calls:

- **Matrix Core Instructions** (MFMA/WMMA): Core matrix multiply-accumulate operations
- **Cross-lane Operations**: Data exchange within a wavefront via `v_perm_b32`
- **Scheduling Barriers**: `llvm.amdgcn.sched.barrier` / `llvm.amdgcn.sched.group.barrier` — used in CK's GEMM implementation to prevent the compiler from reordering manually arranged instruction sequences, reducing data dependencies and compute unit contention

### 1.4 Optimization Methodology Comparison

| Method | Use Case | Representative Projects |
|------|---------|---------|
| C++ Templates + Intrinsics | Extreme performance, precise ISA control | Composable Kernel (CK) |
| DSL + MLIR Lowering | Portability, hierarchical optimization | Triton, FlyDSL |
| Hand-Written Assembly | Lowest-level control, validating architectural understanding | AITER's .co files |

## 2. Triton Optimization Practices on AMD GPUs

### 2.1 AMD Triton Compilation Pipeline

The Triton compiler consists of three modules:

1. **Front-end Module**: Python AST → Triton-IR
2. **Optimizer Module**:
   - Triton-IR optimization (hardware-agnostic: inlining, dead code elimination)
   - Triton-GPU IR optimization (AMD-specific: MFMA layouts, LDS bypass, block ping-pong optimization)
   - LLVM-IR optimization (LLVM O3)
3. **Back-end Module**: LLVM-IR → AMDGCN Assembly → Binary

AMD-specific Triton-GPU IR optimizations include:
- `amd_mfma` layouts: Data layout tailored for AMD's MFMA accelerators
- LDS bypass optimization: Addressing LDS bandwidth issues in MoE kernels
- Block ping-pong optimization

### 2.2 GEAK: AI-Driven Triton Kernel Optimization

GEAK (Generating Efficient AI Kernels) is a framework developed by the community for AI-driven kernel optimization:

**GEAK-OpenAgent** (Triton → Triton):
- **Multi-stage Evolution**: Generates multiple code variants per generation
- **LLM Evaluator**: Multi-dimensional scoring (fusion intelligence, numerical stability, wavefront utilization, etc.)
- **Key Performance**: Achieves an average **3.42x** speedup on TritonBench and an average **7.02x** speedup on ROCm-bench

**GEAK-Optimized Case Study (6.23x Speedup)**:
1. **Swizzling**: Eliminates modulo operations, improves cache locality, and reduces bank conflicts
2. **RMS Norm Optimization**: Reduces memory footprint from a 2D matrix to a 1D vector, replacing `pow()` with direct multiplication
3. **Cache Eviction Policy**: Explicitly sets `eviction_policy="evict_first"`unic for rapid eviction of temporary data

## 3. AMD MoE Kernel Runtime Selection Strategy

The AITER library pre-compiles a large number of hand-written assembly kernels (.co files) for AMD MI300X and other GPUs, automatically selecting the optimal variant at runtime via the `get_heuristic_kernel` function.

### 3.1 Candidate Kernel System

Kernel configurations are driven by CSV files that auto-generate C++ header filesarie. Key parameters for each variant:

| Parameter | Description |
|------|------|
| `subGU_n` | Tile size (N-dimension tiling) |
| `ps` | Whether to use persistent scheduling |
| `tg_num_perCU` | Number of thread groups per CU |
| `smf` | Smooth quantization fusion mode (0=none, 1=full smooth fuse, 2=fc2 only) |
| `vskip` | Whether to enable empty block skipping (padding blocks from MoE sorting) |

### 3.2 Three-Level Heuristic Selection Algorithm

**Step 1: Filtering** — architecture match + vskip/smf consistency + inter_dim divisible by tile

**Step 2: Scoring** (three priority levels, from high to low):

```
tg_num = (inter_dim / subGU_n) * sub_X_cnt // thread group
local_round = ceil(tg_num / num_cu) // execute
empty_cu = local_round * num_cu - tg_num // CU
```

| Priority | Objective | Intuition |
|----------|-----------|-----------|
| 1 (highest) | Minimize local_round | Fewer execution rounds = highest GPU utilization |
| 2 | Minimize empty_cu | Keep more CUs busy |
| 3 | Prefer ps=1 | Persistent scheduling reduces launch overhead |

**Numerical example** (MI300X 304 CU, inter_dim=3584, sub_X_cnt=20):

| tile | tg_num | round | empty_cu | Conclusion |
|------|--------|-------|----------|------------|
| 128 | 560 | 2 | 48 | Eliminated (2 rounds) |
| **256** | **280** | **1** | **24** | **Selected** |
| 512 | 140 | 1 | 164 | Too many idle CUs |

### 3.3 Smooth Quantization Fusion

Fuse the multiply-divide operations of smooth scale into the matrix multiplication ASM kernel, processing directly while data is still in registers/LDS:
- Eliminates the extra element-wise kernel launch overhead
- Avoids additional video memory accesses
- Supports three modes to accommodate different quantization workflows

## 4. RDNA Architecture Optimization Highlights

The RDNA architecture (for gaming/general-purpose computing) differs significantly from CDNA (for AI/HPC), requiring corresponding adjustments to optimization strategies.

### 4.1 Core Architectural Differences

| Feature | RDNA | CDNA |
|---------|------|------|
| Wavefront size | 32 (Wave32) | 64 (Wave64) |
| Matrix instructions | WMMA | MFMA |
| Primary use case | Gaming/rendering/general-purpose | AI training & inference/HPC |
| LDS bank count | 32 | 32 |

### 4.2 Key RDNA Optimization Strategies

**Compute Shader Optimization**:
- Set workgroup size to a **multiple of 64** for best performance
- Maximize write bandwidth: each wave writes **256 bytes** (e.g., 8x8 workgroup writes an 8x8 block)
- Use swizzled layouts to improve bandwidth and cache hit rate
- For full-screen post-processing, CS is **~2% faster** than PS (no need for helper threads to generate triangle edges)

**Shader Optimization**:
- Avoid arc-family functions (atan/acos, etc., 100+ cycles)
- Transcendental functions execute at **1/4 rate**; avoid tan (requires 3 transcendental functions)
- **fp16** can reduce VGPR allocation count
- `readFirstLane` can scalarize values into SGPR
- `gather4` can improve texture fetch speed and reduce VGPR usage (8→4)

**Asynchronous Compute**:
- GCN/RDNA have dedicated compute queues that do not block the graphics queue
- Suitable for masking graphics bottlenecks (z pre-pass, shadow, post-process)
- Small workgroups perform better in async queues (lower resource requirements, more launch opportunities)
- **Note**: Async compute is less effective with export-bound shaders

**Memory Optimization**:
- Create a 256MB large heap and sub-allocate on it
- AMD GPUs have a portion of GPU memory accessible from the CPU (device local + host visible), suitable for updating cbuffers
- Fast clear is **100x faster** than regular clear (requires full-image clear with specific values, e.g., RGBA(0,0,0,0))

## 5. CK (Composable Kernel) Template Optimization Practices

CK is the core framework for optimizing GEMM in the AMD ROCm ecosystem, parameterizing all optimization dimensions through C++ template metaprogramming.

### 5.1 Design Philosophy: "Name as Configuration"

A typical CK kernel name:
```
moe_ck2stages_gemm1_256x32x64x64_1x4_TypeCast_v1_Nswizzle0_Quant0_MulRoutedWeight0_silu_B16_B16_B16
```

| Segment | Meaning |
|---------|---------|
| `256` | BLOCK_SIZE (number of threads in a thread block, = 4 waves) |
| `32x64x64` | MPerBlock x NPerBlock x KPerBlock (tile sizes) |
| `1x4` | MWaves x NWaves (wave arrangement on the tile, 1x4=4 waves) |
| `v1` | Pipeline version (v1=basic, v3=multi-buffered prefetch) |
| `B16_B16_B16` | Input / weight / output data types |

### 5.2 Core Parameter System

**Tile configuration has a significant impact** — for the same problem size, different tile/pipeline combinations can yield up to a 1.45x performance difference. Larger tiles are not necessarily faster, as they may become slower due to register/shared memory/occupancy constraints.

**Wave Arrangement**: MWaves x NWaves describes a two-dimensional grid arrangement of waves on the M-N plane, where each wave processes sub-blocks via MFMA instructions:
```
MXDLPerWave = MPerBlock / (MNPerXDL * MWaves) // M wave MFMA
NXDLPerWave = NPerBlock / (MNPerXDL * NWaves) // N wave MFMA
```

**Pipeline Strategy**:
- **v1**: Basic pipeline / single-buffered, suitable for scenarios with tight shared memory
- **v3**: Multi-buffered prefetch, requires more shared memory, suitable for maximally hiding memory access latency

### 5.3 Block Transfer: Thread-Level Movement from Global → Shared

Approximately 14 template parameters (7 each for A and B, symmetrically configured), with the following core concepts:

- **SliceLengths**: The length of the entire slice covered by this transfer in each dimension
- **ThreadClusterLengths**: How many thread clusters participate in the partition along each dimension
- Satisfies: `SliceLengths = thread_slice_lengths × ThreadClusterLengths` (verified at compile time via static_assert)These parameters directly determine bandwidth utilization and MFMA feeding efficiency. During daily tuning, if you only adjust tiles without modifying the data movement side, you may encounter a bottleneck where "compute is sufficient, but memory becomes the drag."

### 5.4 Template Parameter Management

There are approximately **45** low-level template parameters, but upper-level Python tuning only needs to focus on **7 core parameters** (BLOCK_SIZE, M/N/KPerBlock, MWaves, NWaves, PipelineVersion), with the rest automatically derived by the framework.

Tuning tool: `gemm_moe_tune.py` combined with `profile_fmoe.csv` can perform end-to-end benchmarking to select the optimal configuration.

## 6. MI300X/MI355X Hardware Architecture Essentials

### 6.1 Compute Topology

- **MI300X**: chiplet design, 8 XCDs (304 CUs total), each CU has 4 16-lane SIMDs
- Physically 16 lanes, executing continuously for 4 cycles to complete one wave (64 threads) — a design philosophy of **trading area for quantity**
- Each SIMD has 10 wave slots for context switching (10 × 64 × 4 = 2560 threads)
- Maximum 16 waves per CTA (1024 threads)

### 6.2 Memory Hierarchy

| Name | Size | Cache Line | Policy |
|------|------|-----------|------|
| L1D | 32 KB | 128 B | Write-through |
| L2 | 4 MB (per XCD) | 128 B | Writeback, coherent within XCD |
| LLC (Infinity Cache) | 256 MB total | 64 B | Non-coherent, forced flush to HBM at kernel end |

Kernel termination is a hard boundary for cache coherence — the hardware forces all dirty data from XCDs to be flushed to LLC/HBM.

### 6.3 Register System

| Type | Description | Limit |
|------|------|------|
| SGPR | Scalar registers, shared within wave | Max 104 per CTA |
| VGPR | General-purpose vector registers, per thread | Max 256 |
| AGPR | Matrix accumulation vector registers | Max 256 (shares file with VGPR) |

Spill order: first to AGPR (`v_accvgpr_*`), then to scratch memory (`scratch_store_*`).

Using more than 128 registers requires explicit declaration via `__launch_bounds__` or `__attribute__((amdgpu_waves_per_eu(min, max)))`.

### 6.4 Bandwidth Data

| GPU | Compute | Bandwidth |
|-----|------|------|
| MI300X | 304 CU | 5.2 TB/s (4 IOD × 1300MHz × 1024B) |
| MI355X | 256 CU | 8 TB/s |

Achieving peak bandwidth requires using multiple IODs simultaneously.
