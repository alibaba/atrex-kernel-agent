## Cross-Architecture Pitfall Summary

- **Architecture Identification**: sm_120 ≠ sm_100; do not apply tcgen05 / wgmma / TMEM
- **cp.async Alignment**: `assumed_align=16` unlocks cp_size=128b; val_layout must match copy width (bf16 128-bit → (1,8) not (1,4)); SMEM row stride must be a multiple of 8 bf16
- **Cache Mode Diagnosis**: L1 hit > 50% on streaming workload = L2 false-saturation
- **Multi-Warp Topology Selection**: V-disjoint NOT K-distributed
- **MLIR API Boundary**: `.value` not `.ir_value`; `cpasync.CopyOp` abstract; `vector.extract` unstable
- **TMA Protocol**: Use `pipeline.PipelineTmaAsync` instead of manually writing mbarrier; **producer code must use warp-level guard (`if warp_id == 0:`) rather than thread-level (`if tidx == 0:`)** — `elect_one` inside cute.copy/producer_acquire is a warp-level primitive, a thread-level guard causes intra-warp divergence → deadlock
- **TMA D-1 Unavailable**: The D-1 mode of `PipelineTmaAsync` (all warps produce+consume simultaneously) deadlocks in actual testing — N warps each calling `arrive_and_expect_tx` causes arrival count underflow + N× tx_bytes mismatch; **warp specialization is mandatory**
- **TMA vs cp.async Choice**: TMA is **55% slower** than cp.async for small tiles (≤16KB) with serial dependencies (GDN chunk fwd measured 1.71ms vs 1.10ms); TMA only outperforms cp.async on large tiles (≥64KB) that can be pipelined
- **GDN V31/V113 Dataflow**: `sum_t (K[t,k] * exp_decay[t]) * v_new[t,v]` is equivalent to `sum_t K[t,k] * (exp_decay[t] * v_new[t,v])`; pushing decay to the v_new side removes K-decay scratch, and V113 further reuses `sS`/`sNK_A` B fragments, pushing the no-cache directional final_state path to 0.531-0.533ms
- **Direct Global V Is Not Inherently More Fused**: V30 bypasses shared memory and fetches V RHS directly from global — correct but slower (0.7200ms vs V29 0.6861ms); SASS reveals scalar `LDG.E.U16` falling on the RHS critical path; cp.async staging should still be retained on SM120
- **`cute.recast_tensor`**: Must be inside the dynamic K-loop; hoisting it outside the loop silently breaks multi-tile correctness (v38 NVFP4 GEMM)
- **CUTLASS-Style SF Pack Defaults to 8× Bloat**: `(128, 4)` per-atom layout only carries data in 1/8 of elements; using a compact layout trades out ~21 KB smem to fit BLOCK_K=128 + STAGES=4
- **cute-DSL 4.4.2 `cp.async.bulk` + `PipelineTmaAsync` mbar Incompatibility**: Following the CUTLASS-style "SF via TMA bar" path causes launch failure / garbled output / hang; all three workarounds (mbar wrapper / TMA-for-SF / cp.async.bulk + mbar_ptr) failed validation; cute-DSL 4.4.2's ceiling is 71% of CUTLASS
- **Three-Strike Signal for Multi-Bottleneck Kernels**: NCU single-rule % est. represents architectural ceiling, not wall-time gain. On INT32 MoE data preparation, V8 (warp-agg, 27.52% est.) / V10 (CUB sort, 48.85% est.) / V11-A (warp-specialization, 45.27% est.) — three single-rule attacks all regressed 24-41%; the only survivor V9-A is a 4 KB smem copy (cheap fix). See [cross-architecture meta-rules](../../../ref-docs/nvidia/common/ncu-rule-est-speedup-meta-rules.md) for details
- **`from __future__ import annotations` and `@cute.struct` Are Incompatible**: PEP 563 converts annotations to lazy strings, so cute cannot obtain the actual type object of `MemRange[T,N]` and throws `TypeError`. Any module containing `@cute.struct` must disable future-annotations.
- **`l1tex__t_sector_hit_rate.pct` Includes ld.shared**: On sm_120, L1 + texture share the same physical SRAM; TMA-bypass kernels may still show 30-50% L1 hit; use `sm__inst_executed_pipe_tma` vs `smsp__inst_executed_op_global_ld` to directly verify cache mode
- **vllm.vllm_flash_attn Has No Fast Path on sm_120**: 1645 us vs memcpy ceiling ~99 us = 16× slower; vllm dispatch falls onto the slow generic path on client Blackwell. Do not assume vllm approaches H100 performance on sm_120 when planning attention pipelines
- **Vendor flash_attn.cute Is Not Viable on cute 4.4.2**: cutlass 4.5 introduces `_PipelineOp` private-name + `cutlass.utils.*` re-export + submodule rearrangement; 4.4.2 has none of the three; shim speed cannot keep up with the gap. Wait for cluster cutlass to upgrade to 4.5+ before attempting cute-DSL FA fusion
